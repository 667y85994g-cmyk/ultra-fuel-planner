================================================================================
1. WHAT IT IS
================================================================================

A route-aware fuelling planner for trail and ultra runners. Runs entirely in the
browser (Next.js 15 / App Router). No backend, no database, no authentication.
All state is persisted to localStorage.

The planner takes four inputs:
  - Prior race/run data (for calibration)
  - A GPX route file (for terrain segmentation)
  - A fuel inventory (gels, chews, bars, drink mix, real food)
  - Aid station locations and supplies

It produces:
  - A terrain-aware fuelling schedule (what to eat and when)
  - A carry plan (what to pack between each aid station)
  - A printable race card
  - Hydration and electrolyte guidance
  - Confidence notes and transparency metadata

4. DATA FLOW
================================================================================

User Input → PlannerStore (React Context) → generatePlan() → PlannerOutput

Step by step:

1. User fills in athlete profile (StepAthlete)
   → dispatches SET_ATHLETE, SET_EVENT_META to reducer
   → state saved to localStorage on 500ms debounce

2. User adds prior efforts (StepCalibration)
   → dispatches ADD_PRIOR_EFFORT
   → calibrateFromPriorEfforts() runs to estimate burn rate

3. User uploads GPX (StepRoute)
   → parsed client-side (DOMParser) or server-side (API route)
   → dispatches SET_ROUTE with ParsedRoute
   → segmentRoute() breaks into RouteSegments by gradient

4. User adds fuel inventory (StepFuel)
   → dispatches ADD_FUEL_ITEM / UPDATE_FUEL_ITEM

5. User adds aid stations (StepAidStations)
   → dispatches ADD_AID_STATION

6. User clicks Generate (StepGenerate)
   → runPlanner() in planner-store.tsx:
     a. Calls estimateFinishTime() if no target time set
     b. Scales segment durations to match target/estimated finish
     c. Builds EventPlan object
     d. Calls generatePlan(eventPlan) in fuelling-engine.ts
     e. Attaches finishTimeEstimation metadata
     f. Dispatches SET_PLAN_OUTPUT
   → Navigates to /results

5. CORE ALGORITHMS
================================================================================

5a. TERRAIN SEGMENTATION (segmentation.ts)
─────────────────────────────────────────
Input: ParsedRoute (elevation profile from GPX)
Output: RouteSegment[] with terrain classification

Gradient thresholds:
  - steep_climb:         > 15%
  - sustained_climb:     8% to 15%
  - rolling:             3% to 8%
  - flat_runnable:       -3% to 3%
  - runnable_descent:    -8% to -3%
  - technical_descent:   < -8%

Pace estimation uses Naismith's rule:
  - Base pace: 6.5 min/km on flat
  - Climb penalty: +1.0 min/km per 100m gain
  - Descent factor: 0.85x

Segments are merged when consecutive sections share the same terrain type
and the merged segment would be < 5km.


5b. CARB TARGET ENGINE (carb-target-engine.ts) — Layer 2
─────────────────────────────────────────────────────────
Input: experienceLevel, durationMinutes, racePriority, maxCarbsPerHour
Output: CarbTargetRecommendation (range, working target, rationale)

Core principle: carbohydrate intake targets are determined by RACE DURATION
and RUNNER CONTEXT. They are NOT derived from calories burned.

Why not calories? Caloric expenditure rises with event duration (more fat
oxidation at low intensity). But exogenous carb absorption capacity (~60-90g/hr)
does not increase and may decrease with gut fatigue. Deriving carbs from
kcalPerHour / 12 would push long-race athletes above gut tolerance.

Duration bands (g/hr range) — 6-band structure (updated v2.13):
  <6 hours:    55–90 g/hr
  6–10 hours:  50–80 g/hr
  10–16 hours: 45–70 g/hr
  16–24 hours: 40–65 g/hr   ← key band for 100km / shorter 100-mile runners
  24–36 hours: 35–60 g/hr
  36+ hours:   30–55 g/hr

Ranges shift down gradually — no sudden drop at a single threshold.
The previous single "16+ hours: 30–50 g/hr" band was replaced by three bands
(16–24h, 24–36h, 36h+) to avoid over-conservative targets for the common
14–20 hour race category.

Experience adjustments (g/hr delta to band midpoint):
  elite: +5, experienced: +3, intermediate: 0, novice: -5

Race priority shifts: a_race +3g/hr, completion -2g/hr (within band only)
Gut tolerance ceiling is applied as a hard cap after all adjustments.


5c. CALIBRATION ENGINE (calibration-engine.ts)
──────────────────────────────────────────────
Input: PriorEffort[] + athlete profile
Output: CalibrationResult (burn rate, carb targets, confidence)

Process:
  1. For each prior effort, estimate kcal/hr from duration and distance
  2. If device calories provided, use those (weighted higher)
  3. Average across efforts, weighted by recency and distance similarity
  4. Adjust burn rate for race priority (A-race +10%, completion -5%)
  5. Derive carb target via carb-target-engine (duration-band approach)
     NOTE: burn rate does NOT drive carb targets — it informs timing only
  6. Produce confidence level based on number and quality of efforts

Finish time estimation (hardened v2.27):
  Three-layer protection pipeline:

  Layer 1 — Input validation (isValidEffortForFTE):
    - Rejects efforts with distanceKm < 2.0 or durationMinutes < 10
    - Rejects efforts with implied pace outside 2.5–22.0 min/km
    - Catches malformed data (e.g. stored distanceKm=1 producing 720 min/km pace)

  Layer 2 — Scale ratio filter:
    - FTE_MAX_SCALE_RATIO = 4.0: rejects efforts where
      max(target/reference, reference/target) > 4.0
    - Prevents 160km effort being used as anchor for a 26km target (ratio 6.15)
    - After filtering, prefers same distance class (short/long/ultra), then
      adjacent class, then any remaining within-ratio effort

  Layer 3 — Output sanity check:
    - After scaling + adjustments, verifies implied pace is within [2.5, 22.0] min/km
    - Falls back to buildFallbackFinishEstimate() if output is out of range

  Fallback:
    - buildFallbackFinishEstimate() uses conservativeDefaultPace (derived from
      experience level) when no valid anchor exists or all layers fail
    - Returns low confidence + "no comparable prior effort" explanation

  Distance classes:
    short: < 15 km
    long:  15–40 km
    ultra: > 40 km

  Happy path (when valid effort found):
    1. Select best anchor (closest distance match from filtered candidates)
    2. Scale pace for distance difference
    3. Apply elevation adjustment (+/- for more/less ascent than reference)
    4. Apply race priority adjustment (completion = more conservative)
    5. Output sanity check
    6. Apply 2% conservative bias
    7. Return result with confidence based on distance similarity + data quality


5d. ENERGY MODEL (energy-model.ts) — Layer 3 execution timing only
───────────────────────────────────────────────────────────────────
Uses Minetti (2002) metabolic cost formula:
  Cr = 155.4i^5 - 30.4i^4 - 43.3i^3 + 46.3i^2 + 19.5i + 3.6
  (where i = gradient as decimal)

Used to calculate:
  - caloricBurnRate(gradient, paceMinPerKm, bodyweightKg) → kcal/hr
  - fuellingIntervalMinutes(kcalPerHour) → minutes between fuel events
    Based on ~300 kcal burned → 25g carbs as the timing trigger

IMPORTANT: fuellingIntervalMinutes() determines WHEN to eat, not how much
to target per hour. The carb quantity per event is driven by the athlete's
carbTargetPerHour (set by Layer 2 / carb-target-engine.ts), not by burn rate.

derivedCarbGPerHour() is DEPRECATED — not used in the active pipeline.
Use recommendCarbTarget() from carb-target-engine.ts instead.


5e. FUELLING ENGINE (fuelling-engine.ts)
────────────────────────────────────────
The main plan generator. ~1050 lines. Key components:

TERRAIN_RULES: A lookup table mapping each TerrainType to:
  - preferred fuel types (ordered by priority)
  - avoided fuel types
  - fluid multiplier (e.g., 1.2x on climbs)
  - max items per stop
  - notes for the runner

selectBestFuel(): Multi-dimensional scoring across 11 factors:
  - Type preference match (terrain rules)
  - Sweetness tolerance
  - Late-race tolerance
  - Climb suitability
  - Chewing difficulty on terrain
  - Athlete preferences (noSolids, drinkHeavy, gelLight)
  - Availability (quantity remaining)
  - Solid/liquid ratio guardrail
  - Caffeine budget
  - Sweetness fatigue in late race
  - Exclusion list filtering

Schedule generation:
  - Walks through segments chronologically
  - Calculates caloric burn per segment (energy model)
  - Triggers fuel events at calorie-driven intervals (~300 kcal)
  - Selects best fuel item for each event
  - Tracks running totals (carbs, fluid, caffeine, sodium)
  - Inserts aid station refill events at correct distances
  - Applies early-race / late-race phase adjustments

Section inference (terrain-led, v2.24):
  inferTerrainSections() replaces the old equal-time-bucket approach:
  1. Collect candidate boundaries: terrain-transition points (where broad
     terrain category changes — easy/rolling/climbing/descending) plus
     fatigue-phase waypoints at 25%, 60%, 85% of total distance
  2. Greedy forward pass: accept a boundary only if the resulting section
     is ≥ MIN_SECTION_MINS (65 min) AND enough race remains for at least
     one more MIN_SECTION_MINS section
  3. Split oversized sections (> MAX_SECTION_MINS=150min or > MAX_SECTION_KM=16km)
     at the best terrain transition in the middle third of that section
  4. Build Section objects with accumulated segment durations

  isSignificantTerrainTransition(): flat↔rolling is excluded (minor), all
  other broad-category changes (easy↔climbing, rolling↔descending, etc.) accepted.

  computeSectionCharacter(): returns a human-readable label for each section:
    - Race position ("start", "mid", "late", "final push")
    - Terrain composition (climb density, descent density, runnable proportion)
    - Ascent density (>40 m/km = "big climb", 20–40 = "gradual climb")
    - Labels: "Runnable start", "Long gradual climb", "Technical descent",
      "Late-race rolling", "Final push", "Runnable mid-section", etc.

Carry plan generation:
  - Groups schedule entries between aid stations
  - Sums fluid and carb needs per section
  - Lists specific items to carry
  - Adds practical warnings (heavy section, long gap, etc.)
  - Each section includes sectionCharacter (human-readable label from
    computeSectionCharacter) replacing the old dominantTerrain label

Hydration guidance (v1.6):
  - Derives a practical range from athlete's fluid baseline
  - Adjusts for temperature: warm (>=25C) widens upward, hot (>=30C) more
  - Rounds to nearest 50ml, minimum 100ml spread
  - Returns label + plain-English advice

Electrolyte guidance (v1.6):
  - Tier-based: low / moderate / high
  - Based on temperature, duration, sodium target
  - Returns tier + contextual recommendation text

buildSummary():
  - Aggregates totals (carbs, fluid, caffeine)
  - Calculates coverage score (% of carb target met)
  - Generates fuel format notes (solid/liquid ratio, top fuel types)
  - Attaches hydration and electrolyte guidance

6. PLANNER WIZARD (6 STEPS)

Step 1: Athlete Profile (StepAthlete.tsx)
  - Event name, type, priority, distance, finish time, start time, temperature
  - Experience level (novice/intermediate/experienced/elite)
  - Body weight, carb target, gut tolerance ceiling
  - Fluid target (baseline for hydration range)
  - Sodium target (input for electrolyte tier)
  - Caffeine preference and max
  - Preferences: no solids, drink-heavy, gel-light, low sweetness tolerance

  - Add prior efforts (label, distance, duration, elevation, HR, calories)
  - What worked / what didn't (freeform notes)
  - Shows live calibration result preview

Step 3: Route (StepRoute.tsx)
  - GPX file upload (drag & drop)
  - Elevation profile chart (Recharts)
  - Route stats (distance, ascent, descent, max elevation)
  - Option to skip (flat-course fallback)

Step 4: Fuel Inventory (StepFuel.tsx)
  - Add/edit/remove fuel items
  - Per item: name, brand, type, carbs, sodium, fluid, caffeine, calories
  - Serving size, sweetness score, late-race tolerance
  - Easy-on-climbs / easy-at-high-effort flags
  - Quantity available
  - Pre-seeded with common products

Step 5: Aid Stations (StepAidStations.tsx)
  - Add stations at specific distances
  - Mark what's available (water, sports drink, gels, real food)
  - Custom items list
  - Full refill possible flag
  - Crew access flag

Step 6: Generate (StepGenerate.tsx)
  - Pre-generate summary (athlete targets, route overview)
  - "Generate Plan" button
  - Loading state while engine runs

7. RESULTS PAGE (5 TABS)
================================================================================

Tab 1: Summary (SummaryView.tsx)
  Output follows a deliberate top-to-bottom reading order:
  1. Race overview — event name + estimated race duration
  2. Your fuelling plan — recommended carb range, working target, planned avg,
       plain-English "why" explanation, energy expenditure context (small)
  3. Hydration · Electrolytes · Caffeine — three compact cards
  4. Finish time estimation card (when derived from prior efforts)
  5. On the route — practical fuel format notes (solid/liquid ratio, terrain hints)
  6. Plan reliability card (overall confidence + how personalised the plan is)
  7. Race totals — total carbs, fluid, caffeine, race duration
  8. Fuel inventory match — coverage score + plain-English guidance
  9. What you're using — per-product quantities and carb contributions

  Language principles:
  - "Recommended range" and "Working target" are primary; no burn-rate-derived
    carb numbers shown as targets
  - "Plan reliability" replaces "Plan confidence" for plain-English framing
  - "On the route" replaces "Fuel format rationale"
  - "Fuel inventory match" replaces "Inventory coverage"
  - Energy expenditure shown only as a contextual footnote, not a headline metric
  - All section labels avoid scientific precision framing

  Target terminology (consistent across all views):
  - Recommended range  — the sensible carb range for this runner and race duration
  - Working target     — the target selected for this plan within that range
  - Planned average    — what the generated schedule actually delivers
  Never blur these three; never use "optimal", "precise requirement", or
  "calculated need" to describe any of them.

  Positioning principles (v2.16+):
  - The product is a route-aware fuelling strategy tool, not a nutrition calculator
  - Speak like a coach: practical, grounded, runner-first; not like a lab report
  - Reinforce route differentiation: "where fuelling gets harder on your course",
    "adjusted to what's practical on each terrain", not prescribed rules
  - Trust language to use: "based on current endurance fuelling guidance",
    "working target", "recommended range", "practise in training",
    "often easier to tolerate", "individual tolerance varies"
  - Language to avoid: "optimised", "precise requirement", "scientifically
    calculated", "minimises gut stress", "gut tolerance is highest",
    "gels on climbs, solids on flats" (too prescriptive)

Tab 2: Timeline (TimelineView.tsx)
  - Schedule entries grouped by hour
  - Each entry: time, icon, action, terrain badge, carbs
  - Rationale text for each fuel event
  - Aid station entries highlighted
  - Supports absolute time-of-day display (if race start time set)

Tab 3: Segments (SegmentView.tsx)
  - Per-segment cards with terrain classification
  - Distance, elevation change, estimated duration, effort level
  - Recommended fuel type and rationale
  - Fuel types to avoid on this terrain

Tab 4: Carry Plan (CarryView.tsx)
  - Overview: max fluid to carry (~rounded to 0.5L), max carbs, section count
  - Per-section cards: from/to labels, distance, duration
  - Fluid to carry (~rounded), carbs to carry (exact)
  - Specific items to carry with quantities
  - Refill instructions
  - Warnings (heavy section, long gap, etc.)

Tab 5: Map (RouteMapView.tsx)
  - Leaflet map with OpenStreetMap tiles
  - Route polyline colored by terrain type
  - Fuelling point markers
  - Aid station markers
  - Auto-fits to route bounds
  - Only visible when GPX uploaded

Opens in new browser tab via window.print(). 3-page A4 document.

Layout:
  Page 1 — Race Overview
    - Header: event name, date, distance/ascent
    - Summary metrics strip: duration, carbs/hr, fluid/hr, electrolytes
    - Two-column: Strategy Overview (carb target, primary fuels, drink mix usage,
      hydration/electrolyte notes) + Total Items Required (grouped by type)

  Page 2 — Race Execution
    - Fuelling schedule table: time, km, fuel icon, terrain, carbs
    - Drink mix rows: blue left border; aid station rows: amber background
    - Carry plan grid: from→to, km range, duration, ↑m ↓m, section character,
      fluid target, carb total, items list, warnings

  Page 3 — Course Strategy (only when GPX loaded)
    - Satellite map: Esri World Imagery tiles on 2× DPI canvas (720×400 logical px)
      - Basemap muted: ctx.filter = "saturate(0.30) brightness(0.55) contrast(0.90)"
      - Radial vignette: createRadialGradient (transparent center → 68% black edge)
      - Route: single brand-orange line (#f97316) as visual hero — not terrain-segmented
      - Carry boundaries: white rings (r=9) with inner dot
      - Gel: orange dot #f97316; Chew: blue dot #60a5fa; Bar: violet #a78bfa
      - Drink mix: purple diamond #a855f7 (rotated rect)
      - Aid stations: orange circle #fb923c r=9, white border
      - Start: green #22c55e r=13 "S" label; Finish: red #ef4444 r=13 "F" label
      - Z-order: basemap → vignette → route halo → route line → carry rings →
                 fuel dots → drink mix diamonds → aid stations → start/finish
    - Elevation profile (SVG, inline): area fill, terrain bands, carry boundaries,
      drink mix bands, aid/fuel tick marks, y-axis (m) + x-axis (km) labels
    - Section overview table: section, distance, duration, elevation, terrain, strategy

Design principles:
  - No decimal precision on fluid; ~ prefix throughout
  - No sodium mg/hr values
  - Carbs remain precise (they're inventory-based)
  - Fluid is approximate (ranges, ~ prefix)
  - Electrolytes are contextual (tier labels)
  - Clean enough to glance at during a race
  - Print CSS: @page A4 portrait, 12/15/13mm margins, print-color-adjust: exact
  - window.print() gated behind mapReady state (fires after tiles load or fail)

    Layer A — Section structure first.

v2.2 — Product-Driven Cadence and Event Allocation Fix

  FIX: Inverted the flow. Product reality now determines event size; event size
  determines cadence. The schedule is built bottom-up from actual product specs.

    Never exceeds the product's actual carbsPerServing.

    Lightweight pre-selection used ONLY to determine event size for cadence
    calculation, before the main slot loop runs. Does not consume inventory or
    modify usage counts. Respects terrain rules, phase gates (allowSolids,
    allowChews), and availability. Returns the top-ranked discrete item or null.

      outside the section loop, applied uniformly to all sections.
    New: per-section cadence derived from representative product size:
      1. Call preselectFuelForCadence() for section terrain and phase

v2.3 — Event-Count-First Cadence Inversion
  Core problem fixed: the v2.2 engine still derived cadence first (from product size ÷
  discrete carbs/hr), then generated slots and sliced them. This caused fuelling events
  to cluster at the front of sections, with large gaps later, because generateCadenceSlots()
  places its first event at cadence/2 and repeats, leaving unused tail time when the slot
  list is sliced. Carb delivery per hour was uneven; the schedule did not resemble how
  runners actually fuel.

  NEW PRINCIPLE: Event count is derived first. Cadence is a consequence.

  CHANGED: generateSchedule — section loop cadence logic fully replaced.
    Previous (v2.2) flow:
      1. Derive cadence from (eventSize / discreteCarbsPerHour) × 60
      2. generateCadenceSlots() — first event at cadence/2, then every cadenceMins
      3. Slice slot list to maxEventsNeeded cap

    New (v2.3) flow:
      1. Raw event count = ceil(discreteTargetCarbs / eventSizeG)
         Clamped: minimum 1, maximum sectionHours × 3 (prevents >1 event per 20 min)
      2. Cadence = sectionMins / eventCount
         If cadence < 18 min: reduce eventCount = floor(sectionMins / 18)
         If cadence > 40 min: increase eventCount = ceil(sectionMins / 40)
      3. Delivery validation: ratio = (eventCount × eventSizeG) / discreteTargetCarbs
         If ratio < 0.85: eventCount += 1 (if within cadence floor)
         If ratio > 1.15: eventCount -= 1 (if count > 1)
      4. Generate evenly spaced events:
         interval = sectionMins / eventCount
         For i = 0 to eventCount-1: event at sectionStart + (i+1) × interval
         This places events at equal intervals throughout the section, never clustered.

  UNCHANGED: generateCadenceSlots() remains in the codebase but is no longer used by
    generateSchedule. It is retained as a utility. deriveCadenceMinutes() is still
    exported for use by generateSegmentRecommendations().

    Delivery: 5 × 22g = 110g over 3h = 36.7g/hr (111% of 33g/hr — within 85–115%)

  HYBRID MODEL SUMMARY (v2.3):
    1. Drink mix carbs: strategy.drinkMixSharePct[phase] × sectionTarget, density-capped.
    2. Discrete target: sectionTarget − drinkMixCarbs.

    5. Cadence: sectionMins / eventCount (derived, not set).
    6. Events: evenly spaced at sectionStart + (i+1) × (sectionMins / eventCount).

v2.5 — Cadence Snapping Structural Fix
  Core problem fixed: in v2.4, cadence snapping occurred AFTER event count was
  determined. This created a mismatch: eventCount = ceil(carbs / eventSize) derived
  first, then raw cadence = sectionMins / eventCount, then snap. The snapped cadence
  was larger or smaller than the original raw cadence, so (eventCount × snappedCadence)
  could exceed sectionMins. Events had to be clamped to the section end or silently
  dropped, reducing carb delivery and producing unrealistic gaps later in sections.

  THE FIX: Snap cadence BEFORE deriving event count.
  This guarantees cadence and event count are always algebraically consistent, and
  events can never drift outside section boundaries.

  CHANGED: generateSchedule — section loop cadence pipeline (v2.4 → v2.5)
    Previous v2.4 pipeline:
      1. eventCount  = ceil(discreteTargetCarbs / eventSizeG)  [carb-driven]

      4. events at (i+1) × snappedCadence, clamped to section end   ← drift risk

    New v2.5 pipeline:
      1. rawCadence  = (eventSizeG / discreteCarbsPerHour) × 60  [product-driven]
      2. clamp rawCadence to [18, 40] min
      3. snappedCadence = snapToNaturalCadence(clampedRawCadence)  ← snap first
      4. eventCount  = floor(sectionMins / snappedCadence)          ← count follows
      5. delivery validation: nudge eventCount ±1 if ratio outside 85–115%
      6. events at (i+1) × snappedCadence
         floor() guarantees eventCount × snappedCadence ≤ sectionMins — no clamping.

    Guarantee: no event can fall outside its section. No delivery is silently lost.

v2.6 — Cadence Reconciliation Fix
  Core problem fixed: in v2.5, raw cadence was derived from the product rate
  (eventSizeG / discreteCarbsPerHour) × 60 — independent of how many events
  were actually needed for the section's carb target. This meant the cadence
  calculation was decoupled from carb need: a section needing 2 events and one
  needing 6 events could both produce the same raw cadence if they happened to
  have the same product-rate ratio. After snapping, the final event count
  (floor(sectionMins / snappedCadence)) would silently over- or under-deliver
  relative to the target with no connection to the intended number of events.

  THE FIX: Derive raw cadence from a provisional event count, not from product
  rate. Then snap, then reconcile the final event count from the snapped cadence.
  This two-step approach grounds the rhythm in actual carb need while still
  guaranteeing natural cadences and in-bounds event generation.

  IMPLEMENTATION NOTE — cadence reconciliation pipeline:
    The cadence reconciliation pipeline has four distinct phases:

    Phase 1 — Provisional count (Step A):
      rawCount = ceil(discreteTargetCarbs / eventSizeG)
      This answers: "roughly how many events does this section need?"
      The count is explicitly provisional — it is not the final event count.
      It is clamped to a sanity-checked maximum (3 events per hour).

    Phase 2 — Cadence derivation (Step B):
      rawCadence = sectionMins / rawCount
      This answers: "if we space rawCount events evenly, how far apart are they?"
      Cadence is now grounded in carb need, not product rate.

    Phase 3 — Snap and reconcile (Steps C–D):
      clampedCadence = clamp(rawCadence, 18, 40)
      snappedCadence = snapToNaturalCadence(clampedCadence)
      eventCount_final = floor(sectionMins / snappedCadence)
      Snapping converts mechanical intervals to natural fuelling rhythms.
      floor() guarantees all events stay within section bounds — no clamping,
      no truncation, no silent loss.

    Phase 4 — Delivery validation (Step E):
      deliveryRatio = (eventCount_final × eventSizeG) / discreteTargetCarbs
      If ratio < 0.85: try adding one event (only if it fits within section)
      If ratio > 1.15 and eventCount > 1: remove one event
      The +1 bounds check: (eventCount + 1) × snappedCadence ≤ sectionMins
      This prevents overflow without reintroducing clamping.

    Event generation (Step F):
      Events placed at offset = (i + 1) × snappedCadence from section start.
      floor() in Step D guarantees the final offset is always ≤ sectionMins.

  CHANGED: generateSchedule — section loop cadence pipeline (v2.5 → v2.6)
    Previous v2.5 pipeline:

      G. events at (i+1) × snappedCadence

v2.7 — Fuelling Engine Stabilisation
  Core problem fixed: the v2.2–v2.6 engine produced operationally correct schedules
  in most cases, but still had several paths that could yield implausible outputs:
  (a) cadence could extend to 38–40 min on long low-intensity sections, making
      schedules feel sparse and hard to execute; (b) event sizes were bounded per
      product type but there was no global enforcement — unusual inventory items
      could produce events outside the [15, 45] g operational window; (c) if the
      algorithm produced a constraint violation (overflow, cadence out of range),
      it would propagate silently. The engine was not solving all four operational
      constraints simultaneously as a system.

  THE FIX: Tighten cadence bounds to [15, 35] min, simplify snap values to five
  clean milestones, enforce event size bounds globally, and add a defensive sanity
  check layer that detects and auto-corrects any constraint violations before they
  reach the race card.

  CHANGED: snapToNaturalCadence — new cadence set and range
    Previous: [20, 22, 25, 27, 30, 32, 35, 38, 40] (9 values, range 20–40 min)
    New:      [15, 20, 25, 30, 35]                  (5 values, range 15–35 min)

    Why simplified? Nine values with 2–3 min steps were more granular than necessary.
    Five values at 5-min intervals are easier for runners to internalise and cover
    the full operational range. The jsdoc describes the purpose of each milestone:
      15 min — very frequent (high-intensity / high carb density)
      20 min — frequent top-ups (short race, high carb needs)
      25 min — standard endurance (most mid-distance ultras)
      30 min — relaxed endurance (mixed / food-based fuelling)
      35 min — conservative pacing (long ultra, lower intensity)

    Why max 35 instead of 40? A 38–40 min cadence produces sparse, difficult-to-
    execute schedules. 35 min is the most conservative practical rhythm for ultra
    fuelling; anything beyond this risks energy deficit and cognitive cadence drift.

    Why min 15 instead of 20? Short, high-intensity races or sections with very
    high carb targets may legitimately need 15-min fuelling. Clamping at 20 was
    artificially restrictive for these cases.

  CHANGED: Section loop — cadence clamp range updated
    Previous: Math.max(18, Math.min(40, rawCadence))
    New:      Math.max(15, Math.min(35, rawCadence))
    The clamp now matches the new operational envelope [15, 35].

  CHANGED: Section loop — maxCount cap updated
    Previous: Math.floor(sectionHours × 3)  — cap at 1 event per 20 min
    New:      Math.floor(sectionHours × 4)  — cap at 1 event per 15 min
    Allows up to 4 events per hour when carb targets require it.

  NEW: Global event size bounds enforcement
    After realisticEventSize() returns a per-type value, the result is clamped to
    the global operational range: Math.max(15, Math.min(45, rawEventSize)).
    This is a safety net for unusual inventory items — standard products are already
    within this range from their per-type bounds.

  NEW: Per-section sanity check (step 11 in architecture)
    After Step F (event generation), before slot processing:
    Guard 1 — cadence range: emits SANITY_CADENCE info warning if snappedCadence
      is outside [15, 35]. Should never fire; signals a pipeline bug if it does.
    Guard 2 — event overflow: auto-corrects eventCount if
      eventCount × snappedCadence > sectionMins + 0.5. Emits SANITY_OVERFLOW info
      warning with corrected count. floor() in Step D should prevent this.

  NEW: Layer I — Output sanity pass (post-loop, before schedule.sort)
    Scans all discrete schedule entries. Emits EVENT_SIZE_BOUNDS info warning for
    any entry with carbsG outside [12, 48] g (slightly wider than the [15, 45]
    constraint to avoid false positives on borderline products).
    This catches unusual custom inventory items that slipped through earlier bounds.

  UNCHANGED: Architecture of provisional count → cadence → snap → reconcile (v2.6).
    The core pipeline logic is correct; v2.7 tightens the parameter space and adds
    the sanity layer on top. Section loop Steps A–F are retained and their comments
    updated to reflect the new bounds.

  IMPLEMENTATION NOTE — constraint system:
    The fuelling engine now solves four constraints simultaneously for each section:

    1. Carb delivery ≈ section target (±15% tolerance)
       Enforced by: step A (provisional count from carb need), step E (±1 nudge).

    2. Cadence ∈ [15, 35] min
       Enforced by: step C (clamp to [15, 35]), step D (snap to nearest rhythm).
       Safety net: per-section SANITY_CADENCE warning if violated.

    3. Event size ∈ [15, 45] g
       Enforced by: global clamp after realisticEventSize(), per-type bounds.
       Safety net: Layer I EVENT_SIZE_BOUNDS warning if violated.

    4. All events within section duration
       Enforced by: floor() in step D (eventCount = floor(sectionMins / snappedCadence)).
       Safety net: per-section SANITY_OVERFLOW auto-correction.

    These constraints are solved in a fixed sequence. The system prioritises
    operational realism over mathematical exactness: a 3-hour section producing
    6 events at 30 min (132g delivery at 110% of target) is preferable to 5 events
    at 36 min (an interval that exists nowhere in runner experience).

v2.8 — Cadence/Product Compatibility Fix
  Core problem fixed: in v2.7, the cadence snap set was deliberately tightened to
  [15, 20, 25, 30, 35] and the maximum cadence was hard-capped at 35 min. This is
  correct for most products, but creates an edge case when the representative product
  has a large serving size (e.g. 22g gel) relative to the discrete carb delivery rate
  (e.g. 33g/hr): at a 35-min cadence the algorithm can only deliver 19.25g per
  interval, which is less than the 22g the product provides. The engine would
  technically schedule the gel, but the underlying numbers implied fractional servings
  or under-delivery. The cadence was too tight to fit the product cleanly.

  THE FIX: After snapping cadence (step 7), run a compatibility check. If the product
  is too large to fit at the snapped cadence (within a 10% tolerance), step up through
  an extended cadence ladder [15, 20, 25, 30, 35, 38, 40] until the product fits or
  the maximum 40-min cadence is reached. Values 38 and 40 are only reachable via this
  compatibility path — they are not in the primary snap set and cannot be reached
  through normal cadence calculation.

  COMPATIBILITY CHECK LOGIC (new step 7a):
    cadenceCapacityG = (discreteTargetCarbs / sectionHours) × (candidateCadence / 60)
    Stopping condition: eventSizeG ≤ cadenceCapacityG × 1.1
    The 1.1 factor is a 10% tolerance — a gel can be 10% larger than the theoretical
    capacity without triggering a further step-up. This avoids spurious step-ups for
    borderline fits and matches the operationally sensible output for the worked example.

  WHY 10% TOLERANCE (not 30%)?
    The brief's worked example: 22g gel at 33g/hr, 180-min section.
      At 35 min: cadenceCapacityG = 33 × (35/60) = 19.25g
        With 30% tolerance: threshold = 19.25 × 1.3 = 25g. 22 ≤ 25 → would NOT step up.
        With 10% tolerance: threshold = 19.25 × 1.1 = 21.2g. 22 > 21.2 → step up ✓
      At 38 min: cadenceCapacityG = 33 × (38/60) = 20.9g
        With 10% tolerance: threshold = 20.9 × 1.1 = 23g. 22 ≤ 23 → stop at 38 ✓
    A 30% tolerance would suppress the compatibility check entirely for this input;
    10% correctly triggers the step-up and stops at 38, not 40.

  CHANGED: Section loop — snappedCadence changed from const to let
    Required to allow the compatibility step (7a) to reassign the cadence value.
    `const snappedCadence` → `let snappedCadence`

  NEW: Step C' — Product/cadence compatibility block in section loop
    Positioned between Step C (snap) and Step D (reconcile event count):
    ```
    if (snappedCadence > 0 && sectionHours > 0 && eventSizeG > 0 && discreteTargetCarbs > 0) {
      const discreteCarbsPerHour = discreteTargetCarbs / sectionHours;
      const extendedCadences = [15, 20, 25, 30, 35, 38, 40];
      let extIdx = extendedCadences.findIndex(c => c >= snappedCadence);
      if (extIdx < 0) extIdx = extendedCadences.length - 1;
      while (extIdx < extendedCadences.length - 1) {
        const cadenceCapacityG = discreteCarbsPerHour × (extendedCadences[extIdx] / 60);
        if (eventSizeG <= cadenceCapacityG × 1.1) break;
        extIdx += 1;
      }
      snappedCadence = extendedCadences[extIdx];
    }
    ```
    Guards on all four key values to safely skip the check when any input is zero.
    findIndex(c => c >= snappedCadence) seeds extIdx at the snapped value so the
    compatibility check begins from the already-computed snap, not from the beginning.

  CHANGED: Per-section sanity check — cadence range bound updated
    Previous: Guard 1 checks `snappedCadence > 35` (would fire on any compatibility step-up)
    New:      Guard 1 checks `snappedCadence > 40` (only fires above the maximum allowed)
    Rationale: 38 and 40 are now valid outputs from the compatibility path. The sanity
    check boundary moves to match the new maximum (40 min).

  CHANGED: Architecture comment in generateSchedule updated to v2.8
    Step 7a added (compatibility check description).
    Step 11 updated: cadence range widened from [15, 35] to [15, 40].
    Four-constraints summary: cadence range updated to [15, 40] with note that
    38/40 are only reachable via the compatibility path.
    Header changed from "v2.7 — constrained cadence system" to
    "v2.8 — cadence/product compatibility system".

  UNCHANGED: snapToNaturalCadence() — the snap set remains [15, 20, 25, 30, 35].
    Only the compatibility step (after snapping) can produce 38 or 40.
    The clamp in Step C still uses [15, 35] — unchanged from v2.7.

  EXAMPLE (spec from brief: 3-hour section, 33g/hr discrete, 22g gel):
    Section: 180 min | Discrete target: 99g | eventSizeG: 22g
    v2.7: provisionalCount = ceil(99/22) = 5
          rawCadence = 180/5 = 36 → clamp(36, 15, 35) = 35 → snap → 35

  Two separate briefs addressed in a single release, both fixing variants of the same

  ─── PART A: EVENT-FIRST FUELLING MODEL ───────────────────────────────────────

  Core problem fixed: the v2.2–v2.8 engine allocated drink mix FIRST and subtracted its

    The delivery check now measures only discrete event delivery against the target.

  NEW: 30g/hr effective carb floor
    effectiveCarbsPerHour = Math.max(30, athlete.carbTargetPerHour)
    Very low carb targets (< 30g/hr) over-relied on drink mix and produced sparse discrete

v2.10 — Fuel Rotation and Variety
  Core problem fixed: the v2.9 scoring system selected the highest-scoring product for
  every event in a section. Because gels scored consistently higher than chews or bars
  (particularly due to the long-ultra +25 liquid / -20 solid format bias), plans for
  18-hour races produced long unbroken gel sequences. Schedules looked artificial and
  did not resemble how experienced ultra runners actually fuel.

  THE FIX: Three changes to the allocation model.
    1. Fuel role classification (quick / base / food)
    2. Race-wide rotation tracking with scoring penalties and rewards
    3. Revised format bias that treats chews as a valid secondary fuel

  FUEL ROLE MODEL:
    quick — gels (fast, easy intake between efforts; primary discrete source)
    base  — chews, bars (steady, sustaining; provides variety and palatability relief)
    food  — real food such as banana, potato, flapjack (rare; aid-station only)
    (drink_mix is a continuous section-level source handled separately)

  NEW: fuelRole() helper function (fuelling-engine.ts)
    Classifies an item type as "quick", "base", or "food". Used by selectBestFuel to
    identify role-switching opportunities.

  NEW: Race-wide rotation tracking in generateSchedule
    const recentFuelIds: string[]  — last 2 selected product IDs
    const recentFuelTypes: string[] — last 2 selected product types
    Updated after every successful fuel selection. Persists across sections.
    Passed to selectBestFuel and gap-fill as additional scoring context.

  CHANGED: selectBestFuel signature — two new parameters
    recentFuelIds: string[] = []    (default: empty, no rotation bias)
    recentFuelTypes: string[] = []

  CHANGED: Scoring component 2 — race-distance format bias (revised)
    Previous: blanket isSolid ? -20 : +25 for long ultra (≥10h)
    New (long ultra ≥10h):
      gel:        +20  (primary discrete fuel, unchanged preference)
      chew:        +5  (valid secondary; no longer heavily penalised)
      bar:        -10  (digestive load; only early sections on flat)
      real_food:  -15  (rare; requires aid-station aid)
    New (ultra ≥6h, <10h):
      gel:        +15, chew: +8, bar: -5, real_food: -8

    Why: the previous -20 solid penalty created a 45-point advantage for gels over
    chews. Chews are a realistic and common ultra fuel. The new scoring gives gels a
    15-point lead over chews in long ultras — enough to maintain gel preference while
    letting rotation bring chews into the schedule.

  CHANGED: Scoring component 11 — replaced supply management with rotation scoring
    Removed: remaining = quantityAvailable - used; if remaining <= 2: -20
    (Stock management was misleading in planning mode where quantities are unlimited.)

    New component 11 — Rotation and variety:
      -30 if item.id === last selected ID          (consecutive same product: strong deterrent)
      -12 if item.id === second-last selected ID   (two-ago repetition: mild nudge away)
      -12 if item.type === last selected type      (same type two events in a row)
      +18 if role switch: quick→base or base→quick (reward gel→chew or chew→gel transitions)
      -35 if real_food && !isNearAidStation        (never carry a banana mid-section)
       -8 if real_food                             (additional rarity penalty even at aid)

  SCORE WALKTHROUGH — long ultra with one gel (G) and one chew pack (C):

v2.15 — Drink Mix Over-Allocation Fix (Context-Gated Scheduling)

  Problem: The planner was scheduling one full Maurten 320 serving (80g carbs)
  in every section regardless of terrain, race phase, or section duration.
  For an 18h 100k with a 51g/hr working target this produced:
    Drink mix ×9 = 720g + gels/chews ×24 = ~624g → total 1,344g over 18.5h = 73g/hr
  This was 43% above the working target and exceeded the top of the recommended
  range (40–65g/hr), contrary to the plan being described as conservative.

  Root cause: computeStrategyDrinkMixServings used Math.max(1, Math.ceil(...))
  which forced ≥1 serving in any section where carbsPerServing ≤ sectionTargetCarbs
  (i.e. 80g ≤ ~100g for a 2h mid-race section). The drinkMixSharePct values
  (12–20h band: early 40%, mid 55%, late 65%) were irrelevant — the Math.max(1)
  floor meant drink mix appeared in every single section regardless of share target.

  Fix: Replaced the automatic "always schedule if available" model with a
  context-scoring gate. Three conditions must ALL pass before drink mix is
  scheduled in a section:

    Gate 1 — Context score ≥ DRINK_MIX_MIN_SCORE (3):
      Scored on phase, terrain, section duration, and temperature:
        Phase late:        +3  (solids become harder; drink mix more valuable)
        Phase mid:         +1
        Phase early:        0  (discrete fuelling is easy; drink mix not priority)
        Steep climb:       +3  (heavy breathing, impractical to eat)
        Sustained climb:   +2
        Rolling:           +1
        Technical descent: -1  (hands needed; no fiddly fuelling)
        Flat/other:         0
        Section ≥ 1.5h:    +1  (long enough to justify steady bottle source)
        Section ≥ 2.5h:    +1
        Warm (≥20°C):      +2

    Gate 2 — Minimum gap: ≥90 minutes since last drink mix section.
      Prevents back-to-back scheduling in adjacent short sections.

    Gate 3 — Budget fit: 1 serving must not exceed the full section carb target.
      (Unchanged from before — already existed as a hard guard.)

  Expected result for Race to the Stones profile (18h, flat/rolling, cool):
    Early flat sections (score ~1–2): skip
    Mid flat sections (score ~2):    skip
    Mid rolling sections (score ~3): schedule (borderline, passes gate)
    Late flat/rolling sections (score ~4–5): schedule (most)
    Result: ~4–5 servings × 80g = 320–400g from drink mix
    Discrete fills remaining: ~540–600g from gels/chews
    Total: ~920g over 18.5h ≈ 50g/hr — within working target ±5g/hr ✓

  Secondary changes:
    - drinkMixSharePct values lowered in buildRaceStrategy to reflect selective model:
        12–20h: { early: 25, mid: 35, late: 50 }  (was 40/55/65)

        <6h:    { early: 15, mid: 20, late: 20 }  (was 20/25/20)
    - computeStrategyDrinkMixServings: removed Math.max(1, ...) floor
      (the context gate is now the primary frequency control)
    - generateSchedule signature: added expectedTemperatureC?: number
      (passed through from plan.expectedTemperatureC to enable warm-condition bias)
    - Added getDominantTerrain() and drinkMixContextScore() helpers
    - Architecture comment updated to v2.15

  How drink mix is now budget-limited:
    When drink mix IS scheduled (gate passed), its carbs count against the

v2.16 — Section-Level Fuel Format Budget

  Problem: Even after the v2.15 drink-mix context gate, discrete event selection
  remained gel-dominant. Chews and bars (heavier, more satisfying formats ideal in
  early-to-mid race on flat/rolling terrain) were systematically under-selected
  because the gel base scoring advantage was too large relative to the chew bonus.
  Additionally, allowChews[late] = false for 12-20h races blocked chews for the
  last ~8 hours — too binary for medium ultras.

  Fix: Three-part change to discrete fuel selection:

  1. Section format budget (SectionFormatBudget + computeSectionFormatBudget):
     Each section now computes a soft target for the fraction of discrete slots
     that should be filled with "quick" (gel) vs "base" (chew/bar) formats:

       Phase early, flat/rolling:   quick 48%, base 46%  (near-even split)
       Phase mid, flat/rolling:     quick 56%, base 38%
       Phase early/mid, climb:      quick 65-75%, base 22-32%
       Descent:                     quick 68%, base 28%
       Phase late (any terrain):    quick 82%, base 15%

     These fractions drive scoring bonuses/penalties in selectBestFuel:
       Under target (< 70% of target filled): +12 (quick) or +18 (base)
       At/over target:                        -18 (quick) or -15 (base)

     "base" gets a larger bonus because chews/bars are harder to select against
     the gel default — the asymmetry compensates for terrain and preference biases.

  2. Chew base score +5 → +10:
     In selectBestFuel, the isLongUltra chew bonus was too small to overcome the
     gel advantage on technical terrain. Doubling it to +10 makes chews competitive
     in early-to-mid flat/rolling sections without overriding late-race gel preference.

  3. allowChews[late] = true for 12-20h races:
     The binary exclusion of chews in the late phase of medium ultras was too
     aggressive. The format budget (quick 82% in late sections) and
     lateRaceToleranceScore × 8 already de-prioritise chews naturally. Removing
     the hard gate lets a high-tolerance chew appear occasionally late without
     forcing it.

  Format tracking:
    scheduledQuick and scheduledBase counters are incremented after each
    selectBestFuel call (within the slot loop, per-section). totalSlotsInSection
    is set once at the start of each section's slot loop. Counters reset to 0
    at the start of each section.

  Expected change for Race to the Stones profile (early flat sections):
    Before: all gels in early sections
    After: roughly 1 chew or bar per 2 gels in early-to-mid flat sections;
           late sections remain gel-dominant (quickFraction 0.82)

  CHANGED: src/lib/fuelling-engine.ts — SectionFormatBudget interface (new),
    computeSectionFormatBudget (new), selectBestFuel (4 new params + scoring block #12),
    slot loop (sectionFmtBudget/scheduledQuick/scheduledBase init + both selectBestFuel
    calls updated + format count increment after rotation tracking),
    buildRaceStrategy (allowChews[late] 12-20h: false→true),
    chew base score +5→+10, architecture comment updated to v2.16

v2.18 — Terrain-Aware Pace Distribution + Pre-Climb Fuelling Signals

  Goal: Move from a static fuelling schedule to a terrain-aware fuelling strategy.
  Two connected improvements:
  1. Fatigue-accumulated pacing calibrated to the user's target finish time.
  2. Pre-climb look-ahead that annotates events before significant climbs.

  Part 1: Terrain-Aware Pacing with Fatigue Accumulation

  Background:
    segmentation.ts already uses Naismith's rule to assign terrain-aware durations
    to individual segments (climbs take longer than flats). buildSections accumulates
    these per-segment durations into section fromMinutes/toMinutes, so sections
    already inherit terrain-based timing differences. What was missing:
    (a) Fatigue accumulation — a km 90 climb runs the same pace as a km 5 climb.
    (b) Calibration — Naismith's raw sum often differs from the user's entered finish
        time (e.g. Naismith predicts 10.8h for a 100km flat race but runner targets 18h).
        Previously, totalRaceMinutes was derived from Naismith alone, not the user's target.

  New function: applyFatiguePaceModifiers(segments, targetRaceMinutes)
    1. Computes each segment's position in the race (mid-point km ÷ total km).
    2. Applies a fatigue multiplier based on that position:
         0–25% of race distance: ×1.00  (baseline — early race)
         25–60%:                 ×1.05  (slight fade)
         60–85%:                 ×1.10  (moderate fade)
         85–100%:                ×1.15  (late-race slowdown)
    3. Normalises all durations so their sum = targetRaceMinutes.
       The normalisation preserves the terrain shape (climbs still slower than flats)
       while always respecting the user's finish time prediction.
    4. Updates estimatedPaceMinPerKm to reflect the adjusted duration per km.

  Effect:
    - Early sections are fractionally shorter (fresh pace).
    - Late sections are longer, with climbs amplified by both Naismith and fatigue.
    - Carry plans naturally improve — late sections carry more because they take longer.
    - Caloric burn in deriveTargets now uses fatigue-adjusted pace (more accurate).
    - Total race time always matches the user's entered target finish time.

  Calibration:
    targetRaceMinutes = plan.targetFinishTimeMinutes ?? naisimthSumMinutes
    If the user hasn't entered a finish time, Naismith sum is used (existing behaviour).
    If entered, the entire timing model is normalised to that value — the planner
    now truly uses the user's finish time to set section durations, not just carb targets.

  Race to the Stones example (18h, 100km flat/rolling):
    Before: all sections ~60 min (Naismith flat pace normalised to 18h, no fatigue)
    After:  early sections slightly shorter; late sections slightly longer;
            climbs (if any) disproportionately longer in late race.
    The difference is subtle (±5–15 min per section) but creates a believable,
    runner-familiar pacing curve rather than a flat even distribution.

  Part 2: Pre-Climb Fuelling Signals

  New helper: hasUpcomingClimb(segments, currentKm, lookAheadKm = 1.5)
    Scans forward from currentKm and returns true if a sustained or steep climb
    starts within lookAheadKm on the route. Segments are traversed in order;
    search exits early once beyond the look-ahead window.

  In the slot loop:
    Before each discrete event is pushed to the schedule, hasUpcomingClimb is called
    at slot.distanceKm with a 1.5 km look-ahead. If a climb is detected and the
    current terrain is NOT already a climb (so this is the last easy fuelling
    opportunity before the ascent), the event rationale is prefixed:
      "Fuel now — [terrain] before upcoming climb. [terrain rule rationale]"
    This mimics real race behaviour where experienced runners fuel proactively
    before a climb rather than trying to eat while breathing hard on the ascent.

  Fuel selection is unchanged for pre-climb slots — terrain rules already favour
  gels and chews in high-effort situations, and the current section's terrain rule
  still governs which products are preferred/avoided. The annotation is rationale-only.

  Design notes:
    - The 1.5 km look-ahead is practical: 1.5 km at 9 min/km = ~14 min warning.
      Long enough to act on; short enough to be actionable context, not noise.
    - The check excludes slots already on a climb (terrain === "steep_climb" or
      "sustained_climb") — those events are climb events, not pre-climb events.
    - Technical descent fuelling is already blocked (slotSeg.terrain === "technical_descent"
      → continue); this is unchanged.

  What was NOT changed:
    - segmentation.ts: terrain classification and Naismith pace logic unchanged.
    - Fuel selection scoring: all 12 scoring criteria unchanged.
    - Carb target engine: unchanged.
    - Hydration / electrolyte models: unchanged.
    - Carry plan structure: improved naturally from better section timing.
    - Drink mix context gate: unchanged.
    - UI layout and visual components: unchanged.

v2.20 — Drink Mix Eligibility: Elevation Signals + Narrative Accuracy
────────────────────────────────────────────────────────────────────────────────

Summary:
  Two fixes addressing drink mix appearing too rarely (especially on courses like
  Race to the Stones where terrain classifies as flat_runnable despite meaningful
  ascent), and summary narrative statements contradicting the actual plan when
  drink mix was absent.

Problem 1 — Drink Mix Gate Too Strict on Runnable-Gradient Courses

  Root cause:
    drinkMixContextScore() penalised flat_runnable terrain with -1. Many ultra
    courses (e.g. Race to the Stones) contain long gradual gradients that the
    terrain classifier labels flat_runnable because peak gradient is below the
    sustained_climb threshold (8%), even though sections may accumulate 200–400m
    of ascent and demand steady carb intake. Combined with threshold=4, these
    sections never qualified for drink mix.

  Fix — drinkMixContextScore() in fuelling-engine.ts:

    1. flat_runnable penalty removed (was -1, now 0/neutral).
       Rationale: preventing drink mix on flat sections is now handled by the
       absence of positive signals (low elevation, mid phase scores only +1),
       not by a terrain penalty that also blocks gradual-climb sections.

    2. Elevation burden signals added — new parameters: sectionAscentM, sectionKm:
         ascentPerKm ≥ 15 m/km → +1  (meaningful gradient throughout section)
         ascentPerKm ≥ 30 m/km → +1  (additional for sustained gradient)
         sectionAscentM ≥ 200m → +1  (significant total ascent regardless of distance)

    3. Carb target signal added — new parameter: carbTargetPerHour:
         carbTargetPerHour ≥ 55 g/hr → +1  (drink mix is efficient at high targets)

    Updated typical scores:
      Late/flat (low elev)/2h        = 3 + 0 + 0 + 1         = 4  → passes ✓
      Mid/flat (low elev)/2h         = 1 + 0 + 0 + 1         = 2  → blocked ✓
      Mid/flat (20m/km)/2h/300m asc  = 1 + 0 + 1+1+1 + 1     = 5  → passes ✓
      Late/steep/2h                  = 3 + 3 + 0 + 1         = 7  → passes ✓
      Mid/sustained/2h               = 1 + 2 + 0 + 1         = 4  → passes ✓
      Early/flat/2h                  = 0 + 0 + 0 + 1         = 1  → blocked ✓
      Late/flat/warm/2h              = 3 + 0 + 0 + 1 + 2     = 6  → passes ✓

  Call site updated (generateSchedule):
    sectionAscentM = section.segments.reduce(sum, seg.ascentM)
    sectionKm = section.toKm - section.fromKm
    contextScore = drinkMixContextScore(..., sectionAscentM, sectionKm,
                                        athlete.carbTargetPerHour)

Problem 2 — Summary Narrative Contradicted the Plan

  Root cause:
    buildSummary() contained a hardcoded fuelFormatNotes entry:
      "Solids are scheduled on flatter sections ... Gels and drink mix are
       prioritised on climbs and in the later stages of the race."
    This was appended unconditionally even when drink mix appeared zero times
    in the generated schedule.

    Additionally, strategy.rationale info warnings (generated before the
    schedule) contained references to "drink mix used selectively on climbs"
    even when drink mix was absent.

  Fix 1 — buildSummary() fuelFormatNotes (fuelling-engine.ts):
    Replaced hardcoded string with conditional logic:
      If drink mix appears in fuelEvents → include "Gels and drink mix are
        prioritised on climbs, longer sections, and in the later stages."
      If not → "Gels are prioritised on climbs and in the later stages."

  Fix 2 — Post-filter strategy warnings (generatePlan, fuelling-engine.ts):
    After schedule generation, scan RACE_STRATEGY info warnings. If no drink
    mix item appears in the schedule, strip drink mix clauses from those strings
    using targeted regex replacement before they are returned to the caller.
    This prevents strategy-intent language from contradicting plan output.

v2.24 — Terrain-Led Sectioning Model
────────────────────────────────────────────────────────────────────────────────

Summary:
  Replaced the equal-time-bucket section generator (inferSections) with a
  hybrid terrain-led, aid-station-aware, fatigue-aware model. Sections now
  represent meaningful course features — terrain transitions, fatigue-phase
  shifts, and natural climb/descent groupings — instead of arbitrary ~2h buckets.

Problem:
  inferSections() used Math.round(totalRaceMinutes / 120) to create equal-
  duration time buckets. Race to the Stones produced ~6 sections all at
  ~1h 58min regardless of route character. Section labels said "terrain type"
  but carried no narrative meaning.

New algorithm — inferTerrainSections():

  Stage 1: Collect candidate boundaries
    - Terrain-transition points: where broadTerrain() output changes
      (easy → climbing, rolling → descending, etc.) between consecutive segments
    - Fatigue-phase waypoints at 25%, 60%, 85% of totalDistanceKm
      (boundary forced at each fatigue-phase distance regardless of terrain)

  Stage 2: Greedy forward pass
    - Walk candidates in order, accept a boundary if:
      (a) current section would be ≥ MIN_SECTION_MINS (65 min), AND
      (b) remaining race is ≥ MIN_SECTION_MINS (can form at least one more section)
    - Unaccepted candidates are skipped (section extends further)
    - Final section is whatever remains after last accepted boundary

  Stage 3: Split oversized sections
    - Any section > MAX_SECTION_MINS (150 min) OR > MAX_SECTION_KM (16 km)
      is split at the best terrain-transition candidate in its middle third
    - "Best" = largest broadTerrain category change; if tie, pick midpoint
    - Produces at most one additional section per oversized original

  Stage 4: Build Section objects
    - Each section has: fromKm, toKm, segments[], fromMinutes, toMinutes
    - Duration derived from sum of included segment.estimatedDurationMinutes

  broadTerrain(t: TerrainType) → BroadTerrain:
    flat_runnable, gentle_incline → "easy"
    rolling_mixed                → "rolling"
    sustained_climb, steep_climb → "climbing"
    technical_descent            → "descending"

  isSignificantTerrainTransition(a, b):
    flat ↔ rolling: NOT significant (minor effort change, avoid over-splitting)
    all other broad-category changes: significant

  computeSectionCharacter(sectionSegments, fromKm, toKm, totalKm) → string:
    Derives a human-readable label from:
    - Race position: first 20% → "start", last 10% → "final push" or "final descent",
      last 30% → "late-race", otherwise "" (no position prefix)
    - Terrain composition: if climb share > 40% → climb label with ascent density;
      if descent share > 40% → "technical descent" or "descent";
      otherwise → "runnable" or "rolling"
    - Ascent density: > 40 m/km → "big climb"; 20–40 m/km → "gradual climb";
      < 20 m/km → "rolling" or "runnable"
    - Combined examples: "Runnable start", "Long gradual climb", "Big climb",
      "Technical descent", "Late-race rolling", "Final push", "Runnable mid-section"

sectionCharacter field:
  Added to CarryPlan interface. Displayed in:
  - CarryView.tsx: replaces dominantTerrain label in section metadata row
  - print/page.tsx: replaces terrainLabel(plan.dominantTerrain) in carry grid

Guard rails:
  MIN_SECTION_MINS = 65
  MAX_SECTION_MINS = 150
  MAX_SECTION_KM   = 16

    CarryPlan interface: added `sectionCharacter: string` field
  CHANGED: src/lib/fuelling-engine.ts —

v2.27 — Finish Time Estimator Hardening
────────────────────────────────────────────────────────────────────────────────

Summary:
  Three independent protection layers added to estimateFinishTime() in
  calibration-engine.ts to prevent nonsensical finish time estimates when
  prior efforts contain malformed data or are inappropriate anchors for
  the target race distance.

Failure mode that prompted this:
  User selected "Race to the Stones" prior effort as reference for a 26km
  planning run. Stored distanceKm = 1 (corrupted entry). Engine scaled
  26km from a 1km reference → planning time of 1579 hours. No guard existed.

Layer 1 — isValidEffortForFTE(effort: PriorEffort): boolean
  Rejects an effort from being used as an FTE anchor if:
    - distanceKm < FTE_MIN_EFFORT_KM (2.0)
    - durationMinutes < FTE_MIN_EFFORT_MINS (10)
    - implied pace (durationMinutes / distanceKm) < FTE_MIN_PACE_MIN_PER_KM (2.5)
    - implied pace > FTE_MAX_PACE_MIN_PER_KM (22.0)
  Catches corrupted entries (distanceKm=1, durationMinutes=720 → pace 720 min/km).

Layer 2 — Scale ratio filter
  After layer 1 validation, filter efforts where:
    max(targetKm/referenceKm, referenceKm/targetKm) > FTE_MAX_SCALE_RATIO (4.0)
  A 160km effort is not a valid anchor for a 26km target (ratio 6.15 > 4.0).
  A 100km effort is also rejected for a 10km target (ratio 10.0 > 4.0).
  After filtering, candidates are ranked by distance class match:
    1. Same class (short/long/ultra) — preferred
    2. Adjacent class — fallback
    3. Any within-ratio — last resort
  Best anchor = closest distance match within winning tier.

Layer 3 — Output sanity check
  After scaling + elevation + priority adjustments, verify:
    implied pace of the estimate is within [2.5, 22.0] min/km
  If outside bounds: discard result, fall back to buildFallbackFinishEstimate().

buildFallbackFinishEstimate() — shared fallback
  Extracted from the old "no efforts" path. Uses conservativeDefaultPace
  derived from experience level. Returns low confidence + explanation string
  (e.g. "No comparable prior effort found — estimate based on experience level").

Distance classification:
  classifyDistance(km): "short" | "long" | "ultra"
    short: < 15 km
    long:  15–40 km
    ultra: > 40 km

New constants:
  FTE_MAX_SCALE_RATIO       = 4.0
  FTE_MIN_EFFORT_KM         = 2.0
  FTE_MIN_EFFORT_MINS       = 10
  FTE_MIN_PACE_MIN_PER_KM   = 2.5
  FTE_MAX_PACE_MIN_PER_KM   = 22.0

    isValidEffortForFTE(), buildFallbackFinishEstimate();
    estimateFinishTime() refactored into 9-step hardened pipeline

v2.28 — Event Intent, Simpler Short-Run Logic, and Recovery Guidance
────────────────────────────────────────────────────────────────────────────────

Summary:
  Introduced the EventIntent model — a first-class session type field that
  determines how aggressively race-day fuelling rules apply. Three intents are
  supported: race_day (default, unchanged behaviour), training_run (reduced
  targets, no food restrictions, recovery guidance), and fuelling_practice
  (race-day targets, with recovery guidance).

  Seven behaviour changes:

  1. EVENT INTENT FIELD
     A 3-option "Session type" selector added to the Your Event page before
     event type. Options: Race day / Training run / Fuelling practice.
     Stored in state as EventIntent, passed to EventPlan and propagated
     throughout the engine.

  2. DURATION-AWARE SHORT-RUN CARB REDUCTION (training_run only)
     Training runs apply a carb target reduction based on session length:
       < 5 hours: −8 g/hr from band midpoint
       5–8 hours: −5 g/hr from band midpoint
     Applied in both carb-target-engine.ts (recommendCarbTarget) and
     fuelling-engine.ts (deriveTargets). Fuelling practice sessions receive
     full race-day carb targets.

  3. PRACTICAL END-OF-RUN CUTOFF
     FINISH_CUTOFF_MINS = 15. No discrete fuelling events are scheduled
     within 15 minutes of the estimated finish. Prevents pointless final-
     kilometre gel entries that would not be consumed.

  4. NO LATE-ULTRA FOOD RESTRICTIONS ON TRAINING RUNS
     The late-race solid food penalty (score -= 12) in selectBestFuel() is
     now gated on: isLateRace AND totalRaceHours >= 5 AND eventIntent !== "training_run".
     Training runs of any length can have bars, chews, and real food throughout.
     The buildRaceStrategy() function also overrides allowSolids and allowChews
     to { early: true, mid: true, late: true } for training_run intent.

  5. PRE-RUN FUELLED HANDLING
     New boolean field preRunFuelled on EventPlan. When true, discrete
     fuelling events in the first PRE_RUN_FUELLED_GRACE_MINS (25) minutes
     are suppressed. A "Pre-run fuelled" toggle added to Preferences card in
     the UI. Delays first in-run fuelling after a pre-session meal.

  6. RECOVERY GUIDANCE
     buildRecoveryGuidance() added to fuelling-engine.ts. Returns a
     RecoveryGuidance object for training_run and fuelling_practice intents
     (skipped for race_day and sessions < 45 minutes).
     Guidance covers three windows: within 30 minutes / 1–2 hours / rest of day.
     An optional sodiumNote is added for warm (>=22°C) or long (>=2.5h) sessions.
     Shown in SummaryView.tsx as a green RecoveryCard at the bottom of results.
     Shown in print/page.tsx as a PrintRecoveryGuidance block at the end of
     the Execution Plan page (Page 2).

  7. SESSION TYPE LABELLING THROUGHOUT
     Results page title changes: "Race Plan" / "Training Plan" / "Fuelling
     Practice Plan". A colour badge (blue/purple) appears for non-race-day
     sessions. Print page subtitle changes from "Race Day Nutrition Card" to
     the appropriate session label.

13. DEVELOPMENT NOTES
================================================================================

Running locally:
  cd ultra-fuel-planner
  export PATH="$HOME/node/bin:$PATH"
  npm run dev

Type checking:
  npx tsc --noEmit

Building:
  npm run build

Key design decisions:
  - Rules-based engine (not AI) for trust and transparency
  - Four-layer architecture: Finish Time → Carb Target → Route Execution → Learning
  - Carb targets are duration-band driven (carb-target-engine.ts), NOT calorie-derived
  - Energy model drives fuelling INTERVALS (when to eat), not carb targets (how much)
  - Terrain rules are the main tuning surface (TERRAIN_RULES object)
  - Carbs are precise (inventory-based), fluid is approximate (ranges)
  - Electrolytes are contextual (tiers), not calculated to false precision
  - All state in localStorage — no server round-trips
  - Print page uses inline styles (not Tailwind) for reliable print output
  - Leaflet loaded dynamically (SSR: false) because it requires window

Architecture constraints:
  - No database, no auth, no server state
  - GPX parsing has two paths: client (DOMParser) and server (regex)
  - React 19 with "use client" directives on all interactive components
  - Tailwind v3 (not v4) due to PostCSS compatibility
